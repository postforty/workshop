### 프롬프트 엔지니어링 기법

**1. 명확한 구분자 사용 (Clear Delimiters)**
거대 언어 모델(LLM)에게 요청 사항이나 대화 내용을 제공할 때, 누가 어떤 부분을 담당하고 어떤 문장을 읽어야 하는지 명확하게 구분하여 더 좋은 응답을 얻기 위해 사용됩니다. 사용자가 입력하는 문장에 모호한 부분이 있더라도 응답을 잘 해주는 GPT 4o 모델과 같은 경우에도, 명확한 구분자를 제공하면 훨씬 더 좋은 응답을 얻을 수 있습니다.

*   사용된 구분자의 종류: 하이픈 여러 개, 큰따옴표, 작은따옴표 묶음, 대괄호, 중괄호, 소괄호, 콜론, 콤마, 백틱 세 개 (```).
*   명확한 구분자의 예시로 **샵 두 번(##)**과 **콜론**을 사용하여 카테고리를 정의하는 방법도 소개되었습니다.
*   하이픈을 세 번 이상 여러 번 적어 줄을 끊는 것도 구분이 되는 방법으로 사용되었습니다.

**2. 제로샷, 원샷, 퓨샷 프롬프팅**
채팅 GPT가 대화를 보고 스스로 눈치껏 규칙을 찾아서 응답하도록 유도하는 기법입니다.

*   **제로샷 프롬프팅 (Zero-shot Prompting)**: 예제를 한 번도 보여 주지 않고 요청하는 방식입니다.
*   **원샷 프롬프팅 (One-shot Prompting)**: 예제를 한 번 보여 준 후 응답을 요청하는 방식입니다.
*   **퓨샷 프롬프팅 (Few-shot Prompting)**: 예제를 두 번 이상 여러 번 보여 준 후 응답을 요청하는 방식입니다.
    *   퓨샷 프롬프팅은 특정 형태의 원하는 답이 있을 때 예시를 참고하도록 함으로써 더 정확한 응답을 얻는 데 도움이 됩니다.

**3. 페르소나 패턴 (페르소나 주입)**
ChatGPT와 같은 거대 언어 모델에게 **특정 인격이나 역할, 또는 캐릭터**를 부여하는 것을 의미합니다.

*   사용 방법: "당신은 지금부터 누구누구입니다" 또는 "당신은 지금부터 이러이러한 역할을 수행합니다"와 같은 형태의 문장으로 역할을 부여합니다.
*   효과: 역할을 부여하면 모델이 그 역할에 따라 세계관, 말투, 지식 범위, 관점 등을 다르게 설정하여 응답하도록 유도합니다.
*   예시 역할: 냉철하고 직설적인 동기부여 전문가, 따뜻하고 부드러운 동기부여 전문가, 구석기 시대 사람 등.

**4. 출력 형식 지정 (Output Format)**
요청에 대한 응답을 받기를 원하는 출력 형태를 지정하여 불필요한 설명을 제외하고 원하는 정보만 얻을 수 있도록 합니다.

*   이름만 출력: 추천 이유나 부가적인 설명 없이 항목의 이름만 출력하도록 요청합니다.
*   콤마로 구분: 여러 항목을 콤마(,)로만 구분하여 이름만 나열하도록 요청합니다.
*   JSON 형식: 키(Key)와 값(Value)로 이루어진 JSON 포맷으로 응답을 요청합니다.
*   XML 형식: 데이터를 관리하는 데 사용되는 XML 포맷(태그와 엘리먼트)으로 응답을 요청합니다.
*   순서 없는 목록 (Unordered List): 하이픈 또는 블릿 포인트(점)를 사용하여 순서에 관계없이 항목을 나열하도록 요청합니다.
*   순서 있는 목록 (Ordered List): 1, 2, 3과 같은 숫자를 활용하여 순서가 있는 단계나 방법을 나열하도록 요청합니다.
*   사용자 정의 형식: 제이슨이나 XML과 같은 정해진 형식이 아닌, 사용자가 직접 정의한 특정 기호(예: 중괄호, 콜론)를 조합한 포맷으로 응답을 요청합니다.

**5. 프롬프트 메시지 구성 (Prompt Message Structure)**
복잡한 내용을 다룰 때 응답의 품질을 높이기 위해 프롬프트 메시지를 구조화하는 기법으로, 프롬프트 엔지니어링에서 가장 중요한 부분 중 하나로 언급됩니다. 이 구성 요소가 항상 모두 포함되어야 하는 것은 아니며, 필요에 따라 항목을 추가할 수 있습니다.

*   **지시 사항 (Instruction)**: 모델에게 무엇을 해야 하는지 명확하게 알려 주는 부분입니다 (예: 이 문장을 번역해 주세요, 이러이러한 역할을 수행해 주세요).
*   **제약 사항 (Constraint)**: 모델이 따라야 하는 규칙이나 제한을 명시하는 부분입니다 (예: 100 글자 이내로 작성하세요, 파이썬 언어로 작성하세요).
*   **출력 형식 (Output Format)**: 모델의 답변이 어떤 형태로 나와야 하는지를 정하는 부분입니다 (앞서 설명된 출력 형식 지정과 동일한 개념).
*   **사용자 입력 (User Input)**: 사용자가 모델에게 제공하는 정보가 입력되는 부분입니다 (예: 번역을 해 주어야 할 대상 문장, 리팩토링할 코드 목록).
    *   *참고: 사용자 입력 대신 예시(퓨샷 프롬프팅) 항목을 넣어 구조를 활용할 수도 있습니다.*

### 효과적인 프롬프트 작성을 위한 네 가지 핵심 요소

프롬프트 작성 시 고려해야 할 네 가지 핵심 요소(페르소나, 작업, 맥락, 형식)입니다. 이 네 가지 요소는 효과적인 프롬프트를 작성하기 위해 고려해야 할 주요 영역입니다.

프롬프트 작성 시 네 가지 주요 영역을 모두 사용할 필요는 없지만, 몇 가지 요소를 함께 사용하면 도움이 됩니다. 특히 **동사나 명령이 포함된 '작업(Task)'은 프롬프트의 가장 중요한 구성 요소**이므로 항상 포함해야 합니다.

**1. 페르소나 (Persona)**
이 요소는 AI가 수행할 역할을 지정하거나, 사용자가 누구의 입장에서 프롬프트를 작성하는지 정의합니다.

*   **설명:** 페르소나를 지정하면 Gemini가 출력물의 **어조(tone)**와 **관점(perspective)**을 조정하는 데 도움이 될 수 있습니다. 예를 들어, 프롬프트를 "당신은 선도적인 광고 대행사의 크리에이티브 부서장입니다..."와 같은 문구로 시작하여 역할을 부여할 수 있습니다.
*   **예시 (프롬프트 구성 요소):** "당신은 [업종]의 프로그램 매니저입니다." 또는 "저는 팀장에게 보고하는 행정 담당자입니다."

**2. 작업 (Task)**
이 요소는 AI에게 수행하도록 요청하는 **구체적인 행동**을 정의합니다.

*   **설명:** 작업은 프롬프트의 가장 중요한 구성 요소이며, 항상 **동사나 명령**을 포함해야 합니다. 예를 들어, Gemini에게 요약(summarize), 작성(write), 어조 변경(change the tone), 생성(create) 등 필요한 작업을 알려줄 수 있습니다.
*   **예시 (프롬프트 구성 요소):** "경영진 요약 이메일을 작성하세요".

**3. 맥락 (Context)**
이 요소는 작업을 수행하는 데 필요한 **관련 배경 정보**나 세부 사항을 제공합니다.

*   **설명:** 가능한 한 많은 맥락을 제공하여 Gemini가 요청을 이해하도록 도와야 합니다. 여기에는 관련 프로그램 문서에 대한 세부 정보, 특정 목표 및 대상, 또는 Google Drive에 있는 사용자의 파일 (@file name을 사용하여) 정보 등이 포함될 수 있습니다.
*   **예시 (프롬프트 구성 요소):** "[관련 프로그램 문서의 세부 정보]를 기반으로 [페르소나]에게 보내는"

**4. 형식 (Format)**
이 요소는 AI가 출력물을 제시해야 하는 **구조나 스타일**을 지정합니다.

*   **설명:** 특정 결과를 생성하기 위해 문자 수 제한 또는 생성할 옵션 수와 같은 세부 정보를 프롬프트에 포함하여 제약 조건을 부여할 수 있습니다.
*   **예시 (프롬프트 구성 요소):** "글머리 기호로 제한하세요".

### 네 가지 요소를 모두 사용한 프롬프트 예시

이 가이드에서 제시하는 네 가지 요소를 모두 사용한 프롬프트 예시는 다음과 같습니다:

**영어 원문:**
> You are a program manager in [industry]. Draft an executive summary email to [persona] based on [details about relevant program docs]. Limit to bullet points.

**한글 번역 및 요소 분류:**

| 프롬프트 구성 요소 | 영어 원문 | 한글 번역 |
| :---: | :---: | :---: |
| **페르소나 (Persona)** | You are a program manager in \[industry]. | 당신은 **\[업종]의 프로그램 매니저입니다.** |
| **작업 (Task)** | Draft an executive summary email | **경영진 요약 이메일을 작성하세요.** |
| **맥락 (Context)** | to \[persona] based on \[details about relevant program docs]. | \[관련 프로그램 문서의 세부 정보]를 기반으로 \[페르소나]에게 보내는. |
| **형식 (Format)** | Limit to bullet points. | **글머리 기호로 제한하세요.** |

**전체 한글 프롬프트 예시:**
> **당신은 \[업종]의 프로그램 매니저입니다.** [관련 프로그램 문서의 세부 정보]를 기반으로 [페르소나]에게 보내는 **경영진 요약 이메일을 작성하세요. 글머리 기호로 제한하세요.**

## 유용한 프롬프팅 기법 목록 (Promoting Techniques)

### 1. 샷 프롬프팅 (Shot Prompting: 예시 제공)

모델에게 질문을 던질 때, 정답의 패턴이나 형식을 학습하도록 예시 구문을 함께 제공하는 기법입니다. 예시를 포함하면 LLM이 훨씬 더 잘 이해하고 정확한 답변을 생성할 수 있습니다.

| 기법 | 설명 | 예시 |
| :--- | :--- | :--- |
| **제로샷 프롬프팅 (Zero-Shot)** | 아무런 컨텍스트나 예시 없이 바로 질문을 던집니다. | 영화 리뷰문에 대해 이것이 **긍정적**인지, **중간**인지, 또는 **부정적**인지 판단해 줘. |
| **원샷 프롬프팅 (One-Shot)** | 프롬프트 안에 한 가지의 예시를 포함합니다. | (예시 생략 - 자료에 구체적인 예시 구문 없음) |
| **퓨샷 프롬프팅 (Few-Shot)** | 여러 가지 예시(보통 3가지에서 5가지 권장)를 포함하여 프롬프팅합니다. 예시는 **다양**하고 **퀄리티가 높게** 작성되어야 합니다. | **[프롬프트 내 예시 제공]** 사용자: "스몰 피자 치즈, 토마토 소스, 페퍼로니." 출력 JSON: {"size": "small", "type": "pizza", "ingredients": ["cheese", "tomato sauce", "pepperoni"]}. **[새로운 질문]** 사용자: "페타와 올리브를 넣은 라지 샐러드를 주문했어." 이를 JSON으로 만들어 줘. |

### 2. 역할 및 컨텍스트 지정 프롬프팅

LLM에게 특정 역할, 성격, 배경 지식 등을 부여하여 아웃풋의 스타일과 정확도를 개선하는 기법입니다.

| 기법 | 설명 | 예시 |
| :--- | :--- | :--- |
| **시스템 프롬프팅 (System Prompting)** | 모델의 **가장 기본적인 역할, 기능, 페르소나**를 정의하는 큰 그림(Big Picture) 프롬프트입니다. | 이 리뷰문이 긍정적인지 부정적인지 알려주는 역할을 수행해 줘. |
| **롤 프롬프팅 (Role Prompting)** | LLM 모델의 **성격, 아웃풋 스타일, 보이스** 등 구체적인 역할을 부여합니다. | **너는 투어 가이드**야. 투어 가이드인 것처럼 여행 일정 추천들을 해 줘. |
| **컨텍스트 프롬프팅 (Context Prompting)** | LLM 모델이 답변 생성 시 **참고해야 하는 배경 지식**이나 상황을 제공합니다. | **[컨텍스트 제공]** 너는 지금 80년대 아케이드 비디오 게임에 대한 블로그를 써야 해. **[요청]** 이제 블로그 포스트를 작성해 줘. |

### 3. 사고 과정 제어 기법

LLM이 답변을 생성하는 방식(추론, 논리, 단계)을 제어하여 복잡한 문제 해결 능력을 향상시키는 기법입니다.

| 기법 | 설명 | 예시 |
| :--- | :--- | :--- |
| **체인 오브 사운드 (Chain-of-Thought, COT)** | LLM에게 복잡한 문제(예: 수학 문제, 보고서 작성)를 **단계별로** 생각하며(Step-by-step) 답변을 생성하도록 지시합니다. 이를 통해 정확도를 높일 수 있습니다. COT 기법을 사용할 때는 일반적으로 템퍼러처(Temperature)를 0으로 설정해야 합니다. | 이 수학 문제를 풀 때, **단계별로 스텝 바이 스텝 생각하면서** 한번 문제를 풀어 봐. |
| **스텝백 프롬프팅 (Step-Back Prompting)** | 최종 답변을 생성하기 전에, 먼저 LLM에게 답변에 참고할 만한 **일반적인 혹은 근본적인 개념**을 생성하게 합니다. 그 답변을 컨텍스트로 사용하여 더 특화된 구체적인 답변을 요청합니다. | **[1단계]** 현재 유명한 FPS 게임들이 인기 있는 5가지 스토리라인 세팅을 생성해 줘. **[2단계]** (1단계 결과물을 컨텍스트로 제공하며) 이 중 한 가지를 선택하여 내 게임에 알맞은 스토리라인을 구체적으로 써 줘. |
| **셀프 컨시스턴시 (Self-Consistency)** | 동일한 질문에 대해 **여러 개의 답변을 생성**하게 한 후, 그중 가장 많이 **공통적으로 나온 답변**을 최종 선택(다수결 투표, Majority Voting)하는 기법입니다. 인간의 유머나 감성이 복합적인 **감성 평가** 등 이해하기 어려운 작업에 유용합니다. | 이 이메일이 중요한 내용인지 분류해 줘. 이 과정을 3번 반복하여 가장 많이 나온 답변을 최종 결론으로 도출해 줘. |
| **리액트 (ReAct)** | 모델이 **생각(Thought)**과 **액션(Action/Observation)**을 번갈아 수행하도록 지시합니다. 이는 검색이나 코드를 해석하는 등 **외부 툴을 호출**해야 하는 에이전트 구축에 매우 효과적입니다. | (메탈리카 밴드 멤버의 자녀 수 합산 질문에 대해) **[생각]** 멤버 4명을 찾아야겠다. **[액션]** 멤버 이름 검색. **[관찰]** 4명 확인. **[생각]** 각 멤버의 자녀 수를 찾아 합산해야겠다. **[액션]** 멤버별 자녀 수 검색... |

### 4. 자동 프롬프트 엔지니어링 (APE)

| 기법 | 설명 | 예시 |
| :--- | :--- | :--- |
| **자동 프롬프트 엔지니어링 (Automatic Prompt Engineering, APE)** | 프롬프트 작성이 익숙하지 않을 때, **LLM이 자동으로 프롬프트를 생성**하도록 요청하는 기법입니다. 이는 모델 학습이나 테스트 시 다양한 사용자 요청 유형을 생성하는 데 사용됩니다. | 우리는 티셔츠 웹사이트 챗봇을 학습해야 해. '메탈리카 티셔츠 S 사이즈 1개 구매'를 **사용자가 요청할 수 있는 다양한 방식**으로 예시를 생성해 줘. |

## 좋은 프롬프트 작성을 위한 일반적인 팁

LLM의 답변 퀄리티를 높이기 위해 프롬프트를 작성할 때 염두에 두어야 할 사항들입니다:

*   **간결하고 명료하게 디자인 (Design with simplicity):** 질문을 너무 복잡하게 하면 LLM이 이해하지 못할 수 있으므로, 단순하고 명료하게 프롬프트를 작성해야 합니다.
*   **명령조 사용:** 생각을 말하는 것보다 모델에게 **직접 명령하는 어조**로 작성하는 것이 훨씬 효과적입니다.
*   **액션 동사 활용:** LLM이 수행해야 할 액션을 설명하는 동사(예: **analyze**, **categorize**, **classify**, **contrast**, **compare**)를 많이 사용하는 것이 좋습니다.
*   **명확한 인스트럭션에 집중:** 모델이 **해야 하는 것(Instructions)**에 집중하여 지시해야 하며, **하지 말아야 할 것(Constraints)**에 집중하면 오히려 역효과가 날 수 있습니다.
*   **구체적인 아웃풋 형식 명시:** 답변이 어떤 형식(예: **세 개의 문단**, **JSON 형식**, **XML 형식**)으로 나와야 하는지 구체적으로 명시해 주면, 답변의 일관성을 높이고 할루시네이션(환각)을 줄이며 편차를 줄일 수 있습니다.
*   **다양한 테스트:** 프롬프트 요청 시 질문 형태, 문장 형태, 지시 사항 형태 등 다양한 형태로 요청하고 테스트해 보는 것이 좋습니다.

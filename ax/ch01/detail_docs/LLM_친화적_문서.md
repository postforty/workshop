# LLM 친화적 문서 (LLM-Friendly Documentation)

## 1. 개요
LLM(Large Language Model)이 이해하기 쉽고, 정보를 정확하게 추출할 수 있도록 작성된 문서를 의미합니다. 기계 가독성(Machine Readability)을 높이고 문맥을 명확히 하여 AI의 처리 효율을 극대화하는 것이 목적입니다.

## 2. LLM 친화적 파일 형식 및 예시

### 2.1 Markdown (.md)
구조와 텍스트가 결합된 가장 추천하는 형식입니다. 제목, 목록, 표 등을 명시적으로 표현할 수 있습니다.

```markdown
# 워크숍 계획
## 1. 개요
- **일시**: 2024년 5월 20일
- **장소**: 제주도
```

### 2.2 JSON (.json)
데이터를 키-값(Key-Value) 쌍으로 저장하여 AI가 가장 정확하게 이해할 수 있는 구조화된 데이터 형식입니다.

```json
{
  "workshop": {
    "date": "2024-05-20",
    "location": "Jeju Island",
    "participants": ["Kim", "Lee"]
  }
}
```

### 2.3 Text (.txt)
가장 단순하지만, 구조가 없으면 AI가 이해하기 어렵습니다. 들여쓰기나 기호를 사용하여 논리적 구조를 만들어야 합니다.

```text
[워크숍 계획]
1. 개요
   - 일시: 2024년 5월 20일
   - 장소: 제주도
```

### 2.4 HTML (.html)
웹 문서의 표준으로, 태그를 통해 문서의 구조(제목, 본문, 표 등)를 명확히 파악할 수 있습니다.

```html
<h1>워크숍 계획</h1>
<section>
  <h2>1. 개요</h2>
  <ul>
    <li><strong>일시:</strong> 2024년 5월 20일</li>
    <li><strong>장소:</strong> 제주도</li>
  </ul>
</section>
```

## 3. 작성 가이드 비교

### ❌ 나쁜 예 (인간 친화적, 함축적)
```text
[점심 회식 건]
어제 말한 거기로 예약했어. 
김대리랑 이사님도 오신대. 
12시에 로비에서 봐.
```
> **문제점**: '어제 말한 거기'가 어디인지 알 수 없으며, 날짜가 명시되지 않아 AI가 일정을 등록하거나 장소를 찾을 수 없습니다.

### ✅ 좋은 예 (LLM 친화적, 명시적)
```markdown
# 팀 점심 회식 안내

## 1. 일시 및 장소
- **일시**: 2024년 11월 25일 (월) 12:00
- **장소**: 강남역 1번 출구 '맛있는 파스타' (02-1234-5678)
- **모임 장소**: 회사 1층 로비

## 2. 참석자
- **참석 명단**: 박부장, 김대리, 이사원 (총 3명)
```
> **장점**: 구체적인 날짜, 장소명, 참석자가 명시되어 있어 AI가 캘린더에 등록하거나 식당 정보를 검색하는 등의 후속 작업을 정확히 수행할 수 있습니다.
